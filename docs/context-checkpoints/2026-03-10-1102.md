## Architecture Decisions (Locked)

- `as any` casting for new tables — Supabase TS schema doesn't know about new tables until types are regenerated. Pattern matches rest of codebase (`notifications` uses same approach).
- `WorkflowContext.supabase` is the real typed client; definitions cast to `any` for workflow table ops but use typed client for existing tables (operatives, allocations, etc.)
- Engine uses `createServiceClient()` (bypasses RLS) — workflows run server-side
- Follow-up cron queries active runs first, then per-run targets (avoids cross-table join issues)
- `checkAndCompleteRun()` called after every inbound/upload/followup — marks run completed when no pending/contacted/responded targets remain
